# 需求分析框架参考

## 一、需求分析全流程

```
需求收集 → 需求分类 → 需求建模 → 需求验证 → 需求文档化 → 需求评审
```

---

## 二、需求分类体系

### 按性质分类

| 类型 | 说明 | 举例 |
|------|------|------|
| **功能性需求（FR）** | 系统必须做什么 | 用户登录、订单提交、报表导出 |
| **非功能性需求（NFR）** | 系统如何做 | 响应时间 < 2s、99.9% 可用性 |
| **业务规则（BR）** | 约束和策略 | 折扣不超过 30%、未成年禁止购买 |
| **约束条件（CON）** | 技术或环境限制 | 必须兼容 IE11、使用已有数据库 |
| **假设与依赖（AD）** | 前提条件 | 假设第三方 API 稳定可用 |

### 按优先级分类（MoSCoW）

| 级别 | 含义 | 处理方式 |
|------|------|------|
| **Must Have** | 必须有，缺少则产品失败 | 首期交付 |
| **Should Have** | 应该有，重要但非关键 | 首期尽量交付 |
| **Could Have** | 可以有，锦上添花 | 时间允许则交付 |
| **Won't Have** | 本期不做，未来考虑 | 记录，暂不实现 |

## 三、需求收集技术

### 3.1 用户访谈要点
- 开放式问题先行：「你现在是怎么处理这件事的？」
- 追问痛点：「这里最让你头疼的是什么？」
- 确认频率：「这种情况多久发生一次？」
- 验证理解：「我理解的是…，对吗？」
- 避免引导性问题

### 3.2 用户故事（User Story）
格式：**作为**[角色]，**我希望**[功能]，**以便**[价值]

验收标准（Acceptance Criteria）使用 Given-When-Then：
```
Given [前置条件]
When [执行操作]
Then [预期结果]
```

### 3.3 场景分析
- 正常场景（Happy Path）
- 异常场景（Edge Cases）
- 错误场景（Error Cases）
- 边界条件（Boundary Conditions）

### 3.4 竞品分析
- 功能对比矩阵
- 差异化机会点
- 用户痛点映射

## 四、需求建模工具

### 4.1 用例图（Use Case Diagram）
- Actor（参与者）：人、外部系统
- Use Case（用例）：系统提供的功能
- 关系：关联、包含（<<include>>）、扩展（<<extend>>）、泛化

### 4.2 用例规约（Use Case Spec）
| 字段 | 说明 |
|------|------|
| 用例名称 | 动词+名词，如「提交订单」 |
| 主要参与者 | 触发该用例的角色 |
| 前置条件 | 用例开始前必须满足的状态 |
| 后置条件 | 用例完成后系统的状态 |
| 主成功场景 | 无异常的标准流程 |
| 扩展场景 | 异常或替代流程 |
| 特殊需求 | 性能、安全等非功能需求 |

### 4.3 数据流图（DFD）
- 外部实体（方框）
- 处理过程（圆形/椭圆）
- 数据存储（两条横线）
- 数据流（带箭头的线）

### 4.4 实体关系图（ER 图）
- 实体（矩形）
- 属性（椭圆）
- 关系（菱形）
- 基数：1:1 / 1:N / M:N

### 4.5 状态机图
适用于：订单状态、审批流程、设备状态等
- 状态节点
- 转换条件
- 入口/出口动作

## 五、非功能性需求框架（FURPS+）

| 维度 | 说明 | 典型指标 |
|------|------|------|
| **Functionality** | 功能完整性、安全性 | 权限控制、数据加密 |
| **Usability** | 易用性 | 3步内完成核心操作 |
| **Reliability** | 可靠性 | MTBF、可用性 99.9% |
| **Performance** | 性能 | 并发用户数、响应时间、TPS |
| **Supportability** | 可维护性 | 部署时间、日志完整性 |
| **+约束** | 设计约束、实现约束 | 技术栈、合规要求 |

## 六、需求验证检查清单

### 完整性检查
- [ ] 所有用户角色都有对应需求？
- [ ] 所有业务场景都有覆盖？
- [ ] 异常场景都已处理？
- [ ] 数据边界都已定义？

### 一致性检查
- [ ] 需求间无矛盾？
- [ ] 术语定义统一？
- [ ] 优先级无冲突？

### 可行性检查
- [ ] 技术可实现？
- [ ] 时间资源可支撑？
- [ ] 依赖已确认？

### 可测试性检查
- [ ] 每条需求都有验收标准？
- [ ] 标准可量化、可验证？
- [ ] 无歧义表述？

### 可追溯性检查
- [ ] 每条需求都有唯一 ID？
- [ ] 来源（业务目标/用户诉求）可追溯？
- [ ] 需求与测试用例有映射？

## 七、常见需求陷阱与应对

| 陷阱 | 表现 | 应对策略 |
|------|------|------|
| 需求镀金 | 加入「以后可能用到」的功能 | 严格 MoSCoW 过滤 |
| 需求蔓延 | 范围不断扩大 | 变更控制流程 |
| 隐性假设 | 利益相关者默认对方知道 | 明确追问「你说的X是指？」 |
| 解决方案描述 | 直接说「要做一个按钮」 | 追问「解决什么问题？」 |
| 冲突需求 | 不同干系人需求矛盾 | 优先级排序 + 决策记录 |
| 不可测试需求 | 「系统应该很快」 | 量化：「P99 < 500ms」 |

## 八、干系人管理

### 干系人分析矩阵（权力/利益矩阵）

```
高权力  │  密切管理   │  重点满足  │
        │（满足需求）  │（主要干系人）│
        ├─────────────┼────────────┤
低权力  │  监控       │  保持告知  │
        │（低优先级）  │（保持信息） │
        └─────────────┴────────────┘
           低利益          高利益
```

### 干系人访谈顺序建议
1. 业务决策者（定目标、定优先级）
2. 终端用户（定功能、定体验）
3. 技术团队（定约束、定可行性）
4. 运营/支持团队（定维护需求）

## 九、需求度量指标

| 指标 | 计算方式 | 健康参考值 |
|------|------|------|
| 需求稳定性 | 变更需求数 / 总需求数 | < 20% |
| 需求完整率 | 有验收标准的需求 / 总需求 | > 95% |
| 需求缺陷率 | 因需求问题引发的缺陷 / 总缺陷 | < 15% |
| 需求覆盖率 | 有对应测试用例的需求 / 总需求 | 100% |
