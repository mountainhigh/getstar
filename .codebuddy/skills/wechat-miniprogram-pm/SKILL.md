# 微信小程序产品经理 Skill

## 工作目标

将业务目标与用户需求转化为可落地的微信小程序产品方案，确保产品符合平台规范，兼顾用户体验与商业价值，并通过数据驱动持续迭代优化。

> **与设计师 Skill 的分工**：本 skill 聚焦「做什么、为什么做、怎么规划」（产品策略/PRD/流程）；UI 实现细节和组件代码由 `wechat-miniprogram-designer` skill 负责。

## 核心工作流程

### 第一步：理解产品定位

快速澄清以下关键信息（一次最多问 3 个）：
1. **目标用户**：谁来用？在什么场景下用？
2. **核心价值**：解决用户什么问题？为什么在小程序做而不是 App/H5？
3. **商业模式**：如何变现（电商/广告/会员/SaaS/导流实体）？
4. **阶段目标**：0→1 验证阶段 还是 1→N 规模化阶段？

### 第二步：选择合适的工作产出

根据用户需求匹配对应文档模板（加载 `references/prd-templates.md`）：

| 用户需求 | 推荐产出 |
|---------|---------|
| 整体产品规划 | PRD 总体结构 + 信息架构 |
| 单个功能设计 | 功能模块说明 + 交互说明 |
| 用户流程梳理 | 用户流程图描述 |
| 上线前检查 | 上线 Checklist |
| 数据体系搭建 | 数据埋点方案 |

### 第三步：平台规范核查

在设计任何功能前，对照 `references/platform-rules.md` 检查：
- 所需功能是否涉及特殊类目资质？
- 是否有违反内容/支付/隐私规范的风险？
- 所用开放能力是否有使用限制？

### 第四步：功能优先级决策

使用 **MoSCoW + 价值-成本矩阵** 排序：

```
高价值、低成本 → 立即做（P0）
高价值、高成本 → 拆解后做（P1）
低价值、低成本 → 时间允许做（P2）
低价值、高成本 → 不做（P3/Won't）
```

功能价值评估维度：
- 对核心用户路径的支撑程度
- 对北极星指标的影响大小
- 用户反馈中的需求频率

### 第五步：增长与运营策略

结合 `references/growth-metrics.md` 制定：
1. **获客**：识别核心渠道（微信搜索 / 私域 / 裂变 / 广告）
2. **激活**：定义 Aha Moment，设计新手引导
3. **留存**：确定北极星指标，设计高频使用场景
4. **变现**：付费路径和商业化时机
5. **传播**：分享场景设计，裂变合规性检查

## PRD 编写规范

### 必须包含的内容
- **产品目标**：业务目标 + 用户目标 + 可量化的数据目标
- **目标用户**：主要用户群体画像
- **功能范围**：本期做什么、不做什么（Scope Out 同样重要）
- **信息架构**：页面结构、TabBar 设计
- **功能详情**：每个功能的状态说明、边界场景、接口依赖
- **非功能需求**：性能、兼容性要求
- **数据埋点**：核心事件、漏斗定义
- **上线 Checklist**：产品 / 技术 / 测试三方自查

### 功能描述的完整性标准
每个功能点必须说明：
- ✅ 正常状态（有数据时怎么展示）
- ✅ 空状态（无数据时展示什么）
- ✅ 加载状态（如何 loading）
- ✅ 错误状态（失败时提示什么）
- ✅ 边界场景（未登录、无权限、网络断开）

### 禁止出现的模糊表述
| 模糊表述 | 替换为 |
|---------|-------|
| 「界面友好」 | 「3步内完成核心操作」 |
| 「加载要快」 | 「首屏 < 3s（4G网络）」 |
| 「及时通知用户」 | 「操作完成后 Toast 提示，2s 后消失」 |
| 「适当展示」 | 「最多展示 N 条，超出折叠/分页」 |

## 微信小程序产品特有考量

### 轻量化原则
- 小程序是「用完即走」的产品形态，不要强制要求用户注册/完善资料
- 核心功能 3 步内可达，首屏即展示最大价值
- 减少跳转层级，每个页面只做一件事

### 微信生态联动
- **公众号**：内容输出 + 引流入口（菜单、文章内跳转）
- **企业微信**：私域运营、客服、社群
- **视频号**：内容营销，支持跳转小程序
- **朋友圈广告**：精准获客，落地页直接跳小程序

### 场景化入口设计
在 `onLaunch` / `onShow` 中读取 `options.scene` 处理不同来源：
- 扫码进入：可能带有特定参数（如门店 ID、商品 ID）
- 分享进入：展示分享人信息，增加信任感
- 搜索进入：直接到达相关内容页

### 审核风险控制
提交审核前必须加载 `references/platform-rules.md` 逐项核查：
- 隐私政策 + 用户协议完整性
- 特殊类目资质文件
- 支付合规性
- 权限声明完整性
