# UX 研究与设计流程参考

## 一、以用户为中心的设计流程（UCD）

```
发现(Discover) → 定义(Define) → 构思(Ideate) → 原型(Prototype) → 测试(Test) → 迭代(Iterate)
```

---

## 二、发现阶段：用户研究方法

### 定性研究方法

#### 用户访谈（User Interview）
**何时使用**: 了解用户动机、心智模型、痛点
**访谈大纲模板**:
```
1. 热身（2-3分钟）
   - 介绍自己和研究目的
   - 请用户介绍自己和日常工作

2. 背景探索（5-10分钟）
   - 你通常是如何完成[目标任务]的？
   - 这个过程中最困扰你的是什么？
   - 你最近一次做[目标任务]是什么时候？能描述一下吗？

3. 核心主题（15-20分钟）
   - 针对具体痛点深挖：为什么？还有呢？能举个例子吗？
   - 使用情景引导而非假设性问题

4. 收尾（3-5分钟）
   - 还有什么你觉得重要但没提到的？
   - 你觉得理想的解决方案是什么样的？
```

**访谈原则**:
- 5～8 人可发现约 85% 的可用性问题
- 不引导、不评价、不打断
- 关注行为而非意见（"你做了什么"vs"你觉得什么好"）

---

#### 用户观察 / 情境调查（Contextual Inquiry）
- 在用户真实环境中观察
- "边说边做"协议（Think Aloud）
- 记录实际操作与口头描述的差异

---

#### 焦点小组（Focus Group）
- 6-10 人，1-1.5 小时
- 适合探索认知、态度、偏好
- 不适合可用性测试（从众效应）

---

### 定量研究方法

#### 问卷调查
**Likert 量表设计**:
```
非常同意 / 同意 / 中立 / 不同意 / 非常不同意
```

**常用标准量表**:
- **SUS（系统可用性量表）**: 10 题，评分 0-100，>68 为及格
- **NPS（净推荐值）**: 0-10 分，推荐者(9-10) - 批评者(0-6)
- **CSAT（客户满意度）**: 简单 1-5 分

#### 数据分析
- 漏斗分析（转化率）
- 热力图（点击/滚动）
- 会话录像（FullStory/Hotjar）
- A/B 测试（显著性 p<0.05，置信度 95%）

## 三、定义阶段：分析与洞察

### 用户画像（Persona）模板
```
[姓名] — [职业/年龄]

一句话描述: "我需要______，因为______"

基本信息:
- 年龄/职业/地点
- 技术熟练度（1-5）
- 使用频率

目标:
- 主要目标（核心诉求）
- 次要目标（背景需求）

痛点:
- 最大阻碍 1
- 最大阻碍 2

行为模式:
- 使用场景
- 设备偏好
- 信息获取渠道

一句引言（来自真实访谈）
```

---

### 用户旅程地图（User Journey Map）

| 阶段 | 触点 | 用户行为 | 想法/感受 | 痛点 | 机会点 |
|------|------|---------|---------|------|-------|
| 发现 | 社交媒体广告 | 点击广告 | "这可能有用" | 信息不够清晰 | 优化落地页文案 |
| 注册 | 注册页面 | 填写表单 | "为什么要这么多信息？" | 注册步骤多 | 社交登录 |
| 首次使用 | Dashboard | 查找功能 | "不知道从哪开始" | 无引导 | 新手引导流程 |
| 核心使用 | 核心功能页 | 完成任务 | "终于做完了" | 步骤繁琐 | 简化操作流 |
| 留存 | 邮件/推送 | 回访 | "是否值得继续用" | 价值感不足 | 个性化内容 |

**情绪曲线**: 在旅程图下方绘制情绪起伏（+高兴 / -沮丧）

---

### How Might We（我们如何能…）

将洞察转化为设计机会：
```
问题: 用户在结账时感到焦虑，因为不知道订单状态
HMW: 我们如何能让用户在结账全程保持信心？
HMW: 我们如何能在关键节点主动告知用户进展？
HMW: 我们如何能减少用户对失败的担忧？
```

## 四、构思阶段：创意方法

### 疯狂 8 法（Crazy 8s）
- 将纸对折三次，得到 8 个格子
- 8 分钟内画出 8 个不同方案草图
- 强调数量和多样性，不评判优劣

### 设计工作坊流程（90 分钟）
```
00:00 - 暖场 + 挑战说明 (10min)
00:10 - 独立发散 - 便利贴头脑风暴 (15min)
00:25 - 亲和图归类 (15min)
00:40 - Crazy 8s 草图 (10min)
00:50 - 方案展示 + 投票 (15min)
01:05 - 优选方案深化 (15min)
01:20 - 回顾 + 下一步 (10min)
```

### 优先级排序：Impact vs Effort 矩阵
```
高影响 + 低难度 → 立即做（Quick Wins）
高影响 + 高难度 → 计划做（Major Projects）
低影响 + 低难度 → 可选做（Fill-ins）
低影响 + 高难度 → 不要做（Thankless Tasks）
```

## 五、原型阶段

### 原型保真度选择

| 保真度 | 工具 | 适用场景 | 时间成本 |
|--------|------|---------|---------|
| 纸面原型 | 纸笔 | 早期探索、快速验证概念 | 极低 |
| 低保真线框 | Balsamiq / Wireframe | 结构确认、利益方对齐 | 低 |
| 中保真 | Figma 线框 | 流程验证、团队协作 | 中 |
| 高保真 | Figma / Sketch | 视觉确认、开发交付 | 高 |
| 可交互原型 | Figma Prototype / Framer | 可用性测试 | 高 |

### Figma 原型规范
- 组件化设计（使用 Components & Variants）
- 使用 Auto Layout 确保响应式
- 命名规范：`组件/状态` 如 `Button/Primary/Default`
- 样式统一：Color Styles / Text Styles / Effect Styles
- 原型连线：标注交互类型（点击/悬停/拖拽）

## 六、测试阶段：可用性测试

### 测试计划模板
```
目标: 验证[功能/流程]是否能被[用户类型]顺利完成
参与者: X 名[目标用户]，年龄/技术能力要求
任务:
  1. [任务描述，不提示解决方案]
  2. ...
成功标准:
  - 完成率 > 80%
  - 无辅助完成时间 < X 分钟
  - SUS 评分 > 70
记录方式: 屏幕录制 + 口头协议
```

### 任务设计原则
- 基于真实场景，不提示操作路径
- 正确: "你想购买一件礼物送给朋友，请完成购买"
- 错误: "请点击右上角的购物车"

### 可用性问题严重性评级（Nielsen）
| 级别 | 描述 | 优先级 |
|------|------|-------|
| 0 | 不是可用性问题 | — |
| 1 | 表面问题（不影响任务完成） | 低 |
| 2 | 次要问题（导致延误） | 中 |
| 3 | 主要问题（用户难以完成任务） | 高 |
| 4 | 灾难性问题（用户完全无法完成） | 立即修复 |

## 七、信息架构（IA）

### 卡片分类法（Card Sorting）
- **开放式**: 用户自行分组并命名 → 发现用户心智模型
- **封闭式**: 预设分类，用户归入 → 验证已有架构
- **混合式**: 部分分类预设，部分自定义

### 树形测试（Tree Testing）
- 验证导航结构是否符合用户预期
- 指标：任务成功率、第一次点击正确率、回溯率

### 导航设计原则
```
三次点击原则: 任何内容不超过 3 次点击可达（参考，非铁律）
面包屑导航: 层级 > 3 时提供
返回按钮: 所有非主导航页面都应有返回
搜索: 内容量大时必须提供，支持模糊匹配
```

## 八、设计评审检查清单

### 交互评审
- [ ] 所有状态已设计（默认/悬停/激活/禁用/加载/错误/空态）
- [ ] 流程边界已处理（网络错误/无数据/权限不足）
- [ ] 操作可撤销或有确认机制
- [ ] 信息层次清晰（主次操作区分）

### 视觉评审
- [ ] 遵循设计系统（颜色/字体/间距/圆角）
- [ ] 视觉重心符合用户阅读路径（Z/F 型扫描）
- [ ] 对比度满足无障碍标准
- [ ] 图标含义清晰，必要时配文字标签

### 内容评审
- [ ] 文案使用用户语言（非技术术语）
- [ ] 错误信息给出具体解决建议
- [ ] 空状态有引导操作
- [ ] 加载状态有合理提示
