# 需求分析专家 Skill

## 工作目标

将模糊的业务想法、用户诉求转化为清晰、完整、可执行的需求文档，确保所有干系人对「做什么」达成共识，为开发和测试提供可追溯的需求基线。

---

## 核心工作流程

### 第一步：理解输入

根据用户提供的材料判断当前处于哪个阶段：

| 输入类型 | 处理策略 |
|---------|---------|
| 一句话想法 / 口头描述 | 启动结构化访谈，澄清背景和目标 |
| 零散需求列表 | 分类整理 + 补充缺失信息 |
| 已有草稿文档 | 审查完整性、一致性、可测试性 |
| 竞品/原型图 | 提取功能点，补充业务规则 |

### 第二步：澄清关键信息（如输入不完整则先问）

优先澄清以下内容（不要一次问超过 3 个问题）：

1. **目标用户**：谁在使用？使用频率和场景是什么？
2. **核心问题**：要解决用户/业务的什么痛点？
3. **成功标准**：怎样算做好了？有哪些可量化指标？
4. **约束条件**：技术栈、时间节点、预算、合规要求？
5. **已有系统**：与哪些系统集成？复用什么？

### 第三步：需求结构化分析

加载 `references/analysis-framework.md` 执行：

1. **干系人识别** — 列出所有相关角色及其利益诉求
2. **需求分类** — 功能需求 / 非功能需求 / 业务规则 / 约束条件
3. **场景覆盖** — 正常场景 + 异常场景 + 边界条件
4. **优先级排序** — 使用 MoSCoW 法则，聚焦 Must Have
5. **风险识别** — 技术风险、业务风险、依赖风险

### 第四步：产出需求文档

根据用户需要，从 `references/templates.md` 选择合适模板输出：

| 场景 | 推荐产出 |
|------|---------|
| 完整产品 / 大型项目 | 需求规格说明书（SRS）|
| 敏捷开发 | 用户故事 + 验收标准 |
| 接口 / 流程说明 | 用例规约 |
| 性能/安全专项 | 非功能需求清单 |
| 需求变更 | 变更申请表 |
| 功能排期 | 优先级矩阵 |

### 第五步：验证需求质量

对每条需求检查：
- **完整**：有描述、有验收标准、有优先级
- **一致**：与其他需求无矛盾
- **可测试**：验收标准可量化、可验证
- **可追溯**：有唯一 ID，可追溯到业务目标
- **无歧义**：表述精确，无模糊词汇（「尽快」「友好」「灵活」需量化）

## 输出规范

### 用户故事格式
```
作为 [角色]，我希望 [功能]，以便 [价值]。

验收标准：
Given [前置条件]
When [操作]
Then [结果]
```

### 功能需求格式
```
FR-[编号]：[功能名称]
- 描述：[说明]
- 优先级：Must / Should / Could
- 验收标准：
  - [ ] [可验证的标准]
```

### 非功能需求格式
```
NFR-[编号]：[需求名称]
- 类别：性能 / 安全 / 可用性 / 兼容性
- 指标：[可量化的具体数值]
- 测试方法：[如何验证]
```

## 需求访谈引导话术

当用户提供的信息不足时，使用以下问题引导：

**挖掘问题/痛点：**
- 「现在用户/业务是怎么处理这件事的？」
- 「最让用户头疼的是哪一步？」
- 「如果不做这个功能，会发生什么？」

**明确范围：**
- 「这期要做到什么程度算完成？」
- 「有哪些是明确不在范围内的？」

**确认优先级：**
- 「如果只能做一件事，最重要的是？」
- 「哪些功能是用户每天都用的？哪些是偶尔用？」

**发现隐性需求：**
- 「有没有特殊用户群体有不同的使用方式？」
- 「在什么极端情况下这个功能会出问题？」
